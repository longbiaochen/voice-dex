# 为什么我做了一个只服务 F5 的 Mac 听写工具

我经常在 macOS 上用 ChatGPT Desktop / Codex Desktop 写长中文需求、改代码说明、整理邮件和笔记。真正让我烦的不是“没有语音输入”，而是现有路径总会打断工作流：要么要切应用，要么要维护额外账号/API key，要么转写完还要自己复制粘贴。

所以我做了一个很窄的工具：ChatType。

它的目标不是变成一个全能听写平台，而是把一条很具体的路径做短：

> 光标放到输入框 -> 按 F5 -> 说话 -> 再按 F5 -> 能回填就回填，不安全就放剪贴板。

## 为什么只做一个按键

很多效率工具会越做越完整：悬浮窗、模式切换、语言选择、历史面板、模板、快捷命令。ChatType 这次反过来，只保留一个主动作：

- `F5` 开始录音
- `F5` 停止录音
- 检测到可编辑目标时粘贴
- 检测不到时留在剪贴板

这不是为了显得极简，而是因为我的使用场景非常固定：我正在某个输入框里思考，突然觉得一段话用说的更快。此时最重要的是不要切换上下文。

## 为什么默认不要求 API key

ChatType 面向的是已经在这台 Mac 上登录 ChatGPT / Codex Desktop 的用户。既然本机桌面登录态已经存在，默认路径就尽量复用它，而不是让用户再配置一组 API key、endpoint、模型名。

这样做有一个明显好处：安装好、登录好、授权麦克风和 Accessibility，就可以开始试。

也有一个必须说清楚的边界：ChatType v1 默认依赖本机已登录的桌面会话和上游私有转写路径。它不是一个稳定公开 API 的企业集成，也不应该被包装成“长期保证不变”的 SaaS 能力。

## 为什么粘贴要保守

听写工具最怕的一类问题是：结果很好，但粘到了不该粘的地方。

所以 ChatType 的 paste 行为比较保守。只有检测到当前焦点像是可编辑目标时，才会报告粘贴成功；如果不确定，就把最终文本留在剪贴板，用户手动 `Cmd+V`。

这也是我自己更愿意接受的失败模式：宁愿多按一次粘贴，也不要把一段文字打到错误的窗口里。

## v0.1.2 做了什么

当前公开版本是 `v0.1.2`。

这版加入了 TypeWhisper 术语导入：如果你已经在 TypeWhisper 里维护了工具名、项目名、专有名词，可以在 ChatType 设置里手动导入一份快照。导入后，ChatType 会在 STT 之后做本地确定性术语对齐。

这里仍然没有把默认路径改成“第二轮 AI cleanup”。我希望主路径尽量短、可解释、少漂移。

## 适合谁试

我现在最想找这几类用户试：

- macOS 用户
- 已经使用 ChatGPT Desktop / Codex Desktop
- 经常写中文长 prompt、需求、邮件、笔记
- 认可“安全回填/剪贴板兜底”比“总是自动粘贴”更稳
- 愿意接受 v1 依赖私有桌面转写路径这个现实边界
